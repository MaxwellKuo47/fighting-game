# 傷害測試系統（DPS Lab + 練功房）

> 對象：本專案的人類／AI 共同維護者與玩家。
> 目的：評估角色傷害 **DPS** 與**各技能輸出佔比**，提供兩個前端共用同一套精確歸因：
> 1. **Headless 量測框架** — 給 AI／開發者的單元測試環境（平時不跑，需顯式觸發）。
> 2. **遊戲內練功房** — 給玩家在遊戲中對木人即時測試。
> 狀態：✅ 已實作。`yarn build` 綠 + `yarn test:run` 全綠 + determinism 黃金快照逐位元不變（行為零變動）。

---

## 0. TL;DR

| 我想… | 怎麼做 |
|---|---|
| 跑全角色 DPS 排行榜 + 技能佔比（CLI） | `yarn dps` |
| 在遊戲裡對木人測自己的角色 | 主選單 → 選角 → **🎯 練功房（傷害測試）** |
| 在程式/測試裡量測單一角色 | `import { measureCharacter } from 'test/dpsLab'` |
| 知道「為什麼某招佔比是這樣」 | 看 §3 歸因設計、§4 數字讀法 |

**核心原則**：所有傷害都帶一個 **source slot 標籤**（`basic`/`skill1`/`skill2`/`ultimate`/`summon`/`dot`/`reflect`/`other`），在傷害管線累計成 `perSkill`，故佔比是**實戰連招的精確值**，不是估算。

## 1. Headless 量測框架（`yarn dps`）

純邏輯層（無 DOM/WebGL）對「單一攻擊者 vs 不死木人」跑固定時長連招，量測 DPS 與技能佔比。

### 1.1 執行

```bash
yarn dps                  # = RUN_DPS=1 vitest run test/dps.test.ts --reporter=basic
RUN_DPS=1 yarn test:run   # 連同其他測試一起跑
```

> ⚠️ **平時不跑**：`test/dps.test.ts` 以 `describe.skipIf(!process.env.RUN_DPS)` 閘控。
> 一般 `yarn test:run` 會整包 skip（它是「評估環境」，不是行為回歸測試 —— 後者是 `test/determinism.test.ts` 的職責）。

### 1.2 輸出（範例）

```
=== 角色 DPS 排行榜（sustained / 中性木人 / 20s）===
角色          DPS        總傷  技能佔比（高→低）
─────────────────────────────────────────────
法師        … 大招 45%  普攻 31%  技能1 18%  技能2 6%
…
【法師】(mage)  sustained/neutral靶  20s
  DPS  195.2   總傷 3904   maxHit 130   crit×10
    普攻       1212   31%  ███████
    大招       1700   45%  ███████████
    …
```

### 1.3 API（`test/dpsLab.ts`）

```ts
measureCharacter(charId, opts?) → row      // 量測單一角色
measureAll(opts?) → row[]                  // 量測全名冊（依 DPS 排序）
formatReport(row) → string                 // 單角色技能佔比明細
formatLeaderboard(rows) → string           // 排行榜表
neutralDummyCharId() → string              // 預設中性木人角色
```

**`opts`（皆有預設）**

| 欄位 | 預設 | 說明 |
|---|---|---|
| `seconds` | `20` | 量測時長 |
| `seed` | `0xc0ffee` | 亂數種子（同 seed 完全可重現） |
| `mode` | `'sustained'` | `sustained`=自然 mana/ult/CD；`burst`=`freeMana` 連發（潛在上限） |
| `dummy` | `'neutral'` | `neutral`=非近戰乾淨靶；`boss`=isBoss 靶（繞過近戰 ×0.85、走 boss 傷害修正） |
| `dummyCharId` | 自動 | 自訂木人角色 |
| `charge` | `'full'` | 蓄力技：`full`=蓄滿放、`tap`=點放 |
| `gap` | `PLAYER_RADIUS*2+6` | 攻擊者與木人距離 |

**回傳 `row`**：`{ charId, name, mode, dummy, seconds, total, dps, maxHit, critCount, skillUses, perSkill }`，
其中 `perSkill: { slot, dmg, dps, pct }[]`（依傷害高→低）。

## 2. 遊戲內練功房（玩家用）

### 2.1 進入與操作

- **進入**：主選單 → 角色下拉選單 → **🎯 練功房（傷害測試）**。
- **操作**：移動貼近木人，`J` 普攻 / `K`·`L` 技能 / `;` 大招 / `Space` 閃避（同正常遊戲）。
- **面板（右上角）**：即時 DPS、各技能佔比條、最大單擊、爆擊次數、承受傷害。
  - **DPS 計時自「首次攻擊」起算**：進房後閒置不計入（避免稀釋）；尚未攻擊時顯示「開始攻擊木人以收集數據…」，時間/DPS 皆為 0。
- **按鈕**：
  - **換角下拉** — 即時切換角色（重開測試）。
  - **重置數據** — 歸零累計，並等待下次攻擊重新起算時間（手動）。
  - **木人還手：開/關** — 切換木人是否用 NPC AI 反擊（測承受傷害用）。
  - **離開** — 回主選單。

### 2.2 木人與玩家行為

| 對象 | 行為 |
|---|---|
| 木人樁 | team 2、巨血不死（1e9 HP）、預設**靜止且 pin 在原地**（不被擊退推走）。`還手:開` 時改由 NPC AI 自由移動／反擊。 |
| 玩家 | god mode 滿血不死（持續輸出不中斷；副作用見 §4 限制①）。 |

### 2.3 實作位置

- 模式：`GameMode = 'ffa' | 'boss' | 'training'`（[`types/engine.ts`](src/game/types/engine.ts)）。
- 啟動／維生／推送：[`controller.ts`](src/game/controller.ts) 的 `startTraining` / `logicTick` 訓練區塊 / `emitTrainingStats`。
- UI：[`TrainingOverlay.tsx`](src/components/TrainingOverlay.tsx)，由 `App` 訂閱 `trainingStats` 事件渲染。
- controller API：`startTraining(charId?, {retaliate?})` / `resetTrainingStats()` / `setTrainingRetaliate(on)` / `quitTraining()`。

## 3. 歸因設計（維護者）

### 3.1 資料流

所有傷害匯流到 [`dealDamage(state, target, amount, attackerId, opts)`](src/game/entities/damage.ts)，`opts.source` 決定歸因桶；
[`recordDamage`](src/game/entities/stats.ts) 累計 `perPlayer[id].perSkill[source] += amount`；
[`summarizeDps(state, id)`](src/game/entities/stats.ts) 整理成 DPS 摘要（lab 與練功房共用）。

### 3.2 source 解析順序（`dealDamage` 內）

1. 攻擊者是召喚物／召喚 → **`summon`**（並歸功主人）。
2. `opts.source`（deferred 實體建立時 stamp 的標籤）。
3. 攻擊者 `_srcSlot`（同步施放期由 `executeAction` 設定）。
4. `opts.dot` → **`dot`**；皆無 → **`other`**。

### 3.3 兩種標記機制

| 傷害型態 | 機制 |
|---|---|
| **同步**（melee / 衝鋒命中 / multiblink / 大部分客製技） | [`executeAction`](src/game/actions/executor.ts) 進入時設 `caster._srcSlot = slot`（save/restore 支援巢狀），離開還原。dealDamage fallback 自動接住，**handler 不必改**。 |
| **Deferred**（投射物 / 區域 / DoT / 衝鋒 / 躍擊 / 汲取 / 連射 / 留痕 / 星環箭暴 / 武士居合） | 建立時把 `ctx.source` stamp 到實體（`makeProjectile`/`makeZone` 的 `srcSlot`、`charge`/`leap`/`channel`/`barrage` 物件、DoT effect 經 `applyEffectFrom` 注入），解析時傳 `{ source: entity.srcSlot }`。 |

### 3.4 為什麼行為零變動

只「加欄位」，不改任何傷害數值或分支；determinism 黃金快照的 `digestState` 只讀**效果的 key** 與**實體 `.length`**，不讀 `srcSlot` → 快照逐位元不變（已驗證）。`recordDamage` 在 `state.stats` 不存在時為 no-op，故正常對戰零成本。

## 4. 數字怎麼讀（限制與正確解讀）

這些是「1v1 木人 DPS」方法論的**固有特性，不是 bug**，但解讀時要知道：

| # | 限制 | 影響 | 對策 |
|---|---|---|---|
| ① | 攻擊者**滿血不死** | `lowHpBonus`/殘血加成不觸發、hpCost 技恆可放 → 狂戰士類**偏低估** | 看殘血流時手動參考；可日後加固定血量% 選項 |
| ② | **1v1 量不到團隊增益** | warsong(需隊友)/protect/ally buff = 0 → **支援/治療被低估** | 排行榜本質偏輸出職 |
| ③ | **召喚物歸因粗** | 寵物傷害全進 `summon` 桶，且受 AI uptime/走位影響 → 召喚師噪音大 | 看趨勢不看絕對值 |
| ④ | **增益技顯示 0%** | overdrive/mark 等「放大別招」自身佔比=0（價值反映在被放大的技） | 別據佔比判斷增益技強度 |
| ⑤ | 木人**身分≠真實目標** | neutral 靶無近戰 ×0.85、無相位減傷 → 絕對值是「對乾淨靶」 | 用 `dummy:'boss'` 對照闖關實感 |
| ⑥ | 命中率**理想化** | lab pin 木人+自動瞄準≈100% 命中 → 慢速彈/窄 AoE 實戰更低 | 練功房用真實手動瞄準對照 |
| ⑦ | **charge 策略單選** | `full` vs `tap` 大幅改變蓄力技 DPS | 兩種都跑來看區間 |
| ⑧ | **sustained 受時長/ramp 影響** | combo/fury/iaido 等會隨時間 ramp | 固定用 20s 當基準比較 |

**建議讀法**：`sustained/neutral` 比輸出職持續 DPS；`burst` 看爆發上限；`boss` 靶看闖關實感；
**技能佔比**用來看「輸出結構（哪招在 carry）」，不要拿來評支援/增益技。

## 5. 擴充指南（新增角色時）

| 情況 | 要做什麼 |
|---|---|
| 用標準 handler（projectile/zone/melee/charge/leap/channel/barrage/grapple/blink/buff/dash/summon） | **零工作**：handler 已從 `ctx.source` stamp，自動精確歸因。 |
| 同步的客製傷害（在 `executeAction` 當下就 `dealDamage`） | **零工作**：`caster._srcSlot` fallback 自動接住。 |
| **自走 tick 的客製 deferred 傷害**（像 star-orbit 箭暴 / samurai 居合：施放後存狀態、之後幾幀才結算） | 在持續狀態物件存 `srcSlot: ctx.source`，於 tick 結算的 `dealDamage`/`applyEffectFrom` 傳 `{ source: state.srcSlot }`。**忘了就會落入 `其他`（誠實標示，非錯誤）。** |

驗證：`yarn dps` 看該角色 `其他` 佔比是否異常偏高；`test/dps.test.ts` 的「`perSkill` 加總 == 總傷」不變量會抓歸因遺漏/重複。
